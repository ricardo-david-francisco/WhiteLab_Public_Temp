# Architecture Decision Records (ADRs)

> **Why this folder exists.** The repository's running summary of
> unified contributions emits *critiques* — proposals to change the
> architecture. Some are accepted, some are rejected, some are adapted
> into a different shape. ADRs are the immutable, dated record of those
> outcomes. They are the *human* answer to "why did we do it this way?"
> three years from now.

## Format

Each ADR is a single Markdown file named `ADR-NNNN-<slug>.md` and follows
the [Michael Nygard format](https://github.com/joelparkerhenderson/architecture-decision-record).
The minimal sections are: **Status**, **Context**, **Decision**,
**Consequences**. The YAML front-matter is mandatory — it is what the
critique importer (`tools/critique/sync-critique-findings.sh`) reads to
auto-close superseded critique findings.

```yaml
---
id: ADR-0007
status: Accepted          # Proposed | Accepted | Superseded | Deprecated
date: 2026-05-06
supersedes: []            # zero or more ADR ids this one replaces
related-issues: []        # GitHub issue numbers this decision addresses
related-critique: []      # anchors in unified-critique.md, e.g. "#3-hardening-…"
tags: []
---
```

## Lifecycle

1. A critique finding lands in `unified-critique.md`. The importer creates
   an Issue labelled `critique`.
2. You either:
   * **Accept** → file an RFC in `docs/contributions/pending-RFCs/`,
     implement it, close the issue with the merging PR. No ADR strictly
     required (the merged code is the record).
   * **Reject** or **Adapt** → write an ADR here. List the critique
     anchor in `related-critique:`. Close the issue with body text
     `Adopts ADR-NNNN`.
3. The next run of the critique importer reads the ADR's
   `related-critique:` list and:
   * marks the corresponding finding as *Adopted* in the issue body,
   * appends an entry to the auto-generated **Decisions** appendix at
     the top of `unified-critique.md`.

## Why an ADR (and not just a closing comment)?

Because comments are tied to the platform. ADRs are plain Markdown
checked into the repo: if you ever leave Copilot, GitHub, or this AI
pair-programmer entirely, the *why* of every architectural decision
travels with the code. The ADR folder is the single most portable piece
of institutional memory in the project.
