# WhiteLab 2.0 — Fortress Design

> **Purpose.** Turn this repository into the **single source of truth and
> control plane** for the home infrastructure described in
> [`docs/research/01-infrastructure-bible.md`](../research/01-infrastructure-bible.md).
> "Fortress" because every change to OPNsense, Proxmox (N95 + N305) and
> the Omada controller flows through a hardened pipeline whose two
> non-negotiable rules are:
>
> 1. **No SSH on the targets, ever.** Every read and write uses the
>    target's native HTTPS API, scoped to the smallest possible
>    privilege.
> 2. **No plaintext secret ever leaves a trusted machine.** GitHub,
>    Copilot, Snyk and any third-party tool see only deterministically
>    anonymized artefacts. Re-hydration happens *only* on the home-side
>    Fortress Agent, which holds the secret map under an `age`-encrypted
>    vault.
>
> This document is the contract. Every directory layout, workflow,
> tool, and runbook in WhiteLab 2.0 derives from it.

## 1. Goals & non-goals

### 1.1 Goals

* Treat WhiteLab as a **declarative source of truth** for:
  * OPNsense configuration (`config.xml`, firewall rules, aliases, DHCP, DNS, services, captive portal, WireGuard, hardening tunables).
  * Proxmox host configuration (sysctl, cluster firewall, Fail2Ban, 2FA, network bridges, unattended-upgrades, masked SSH units).
  * LXC container definitions (golden images, per-CT `pct` flags, mount entries, network, audit scripts).
  * Omada controller configuration (port profiles, wired networks, SSID-to-VLAN mapping, controller settings).
  * Reverse-proxy / Caddy configuration on LXC 100.
* Make every change to those targets a **pull request** with:
  * Static analysis (lint, schema, OPA policies).
  * Vulnerability scanning (Snyk, Trivy).
  * Anonymization invariant check (no raw secret slipped through).
  * One human approval.
  * One TOTP-confirmed apply (defense in depth — GitHub compromise alone cannot ship a change).
* Apply changes **without SSH** by talking to native HTTPS APIs from a single hardened agent on the home network.
* Keep the agent and the apply path inside the **Zero-Trust** model defined by the bible (correct VLAN, Tailscale or WireGuard, scoped API tokens).
* Give the user a **DevOps-style workflow** without requiring DevOps experience: clone, branch, edit, PR, approve, observe.
* Audit every LXC template and every dependency with **Snyk**, **Trivy**, **Lynis**, and a small **OPA** policy bundle before promoting it to "golden" status.

### 1.2 Non-goals (initially)

* Multi-user / team workflows — only the repo owner approves and applies.
* Public-internet exposure of any service. Tailscale + WireGuard remain the only remote-access paths.
* Migrating off Proxmox / OPNsense / Omada / unRAID. The bible's components are fixed.
* Real-time configuration sync. Apply is on-demand and PR-gated, not continuous.

## 2. Threat model

### 2.1 Adversary classes

| Class                     | Capability                                                                   | We must survive ...                                                                                                       |
| ------------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `A1 — Public leak`        | The repo becomes public (rogue admin, GitHub bug, account takeover).         | Zero exploitable secret in any commit, branch or tag. Zero usable internal hostname / public IP / MAC / tailnet ID.       |
| `A2 — Copilot / AI`       | Copilot or any AI-side indexer reads every committed byte.                   | AI sees only anonymized artefacts; placeholder map never leaves the home machine.                                         |
| `A3 — Compromised laptop` | Working copy + GitHub PAT lifted from a dev laptop.                          | PAT is fine-grained (single repo, no admin), no apply privilege; cannot push to protected branches; cannot trigger apply. |
| `A4 — Compromised CI`     | Attacker gets `GITHUB_TOKEN` inside Actions.                                 | CI cannot apply. CI has no production credentials. CI only runs read-only scans.                                          |
| `A5 — Compromised vendor` | Snyk / Trivy / a Python dep is malicious.                                    | Sandboxed CI (read-only mounts, no network egress beyond the API list). Vendor outputs cannot influence apply.            |
| `A6 — Compromised LXC`    | An attacker takes one LXC (e.g. 101 Omada).                                  | LXC is unprivileged + UFW deny-by-default; cannot reach Fortress Agent's vault; cannot reach API tokens.                  |
| `A7 — Lost agent host`    | The agent LXC is destroyed (host fire, drive failure).                       | Agent state is deterministic from the encrypted vault + repo; rebuild from a paper recovery code in <30 min.              |

### 2.2 Assets & where they live

| Asset                                                            | Allowed locations                                                | Forbidden locations                |
| ---------------------------------------------------------------- | ---------------------------------------------------------------- | ---------------------------------- |
| OPNsense root password                                           | OPNsense host, offline vault                                     | git, AI, CI                        |
| OPNsense API key+secret (read-only)                              | Fortress Agent vault, offline vault                              | git, AI, CI                        |
| OPNsense API key+secret (write)                                  | Fortress Agent vault (loaded only during apply), offline vault   | git, AI, CI, agent disk persistent |
| Proxmox API tokens (per-token scopes)                            | Fortress Agent vault                                             | git, AI, CI                        |
| Omada controller admin token                                     | Fortress Agent vault                                             | git, AI, CI                        |
| WireGuard private keys                                           | OPNsense host (for the server), each peer device, vault          | git, AI, CI                        |
| WireGuard public keys + PSKs                                     | git **only after age encryption with SOPS**                      | git in plaintext                   |
| TLS certs (Tailscale-issued)                                     | LXC 100 disk, fortress vault for rotation                        | git in plaintext                   |
| Anonymization map (`anonmap.age`)                                | Fortress Agent vault, offline vault                              | git, AI, CI, dev laptop persistent |
| Real MAC addresses, public IPs, tailnet name, internal hostnames | OPNsense host, vault, agent runtime memory                       | git, AI, CI, dev laptop persistent |
| Snyk API token                                                   | `.sandbox/state/snyk` (gitignored), GitHub Actions secret        | git plaintext                      |
| GitHub fine-grained PAT (read+PR write)                          | dev laptop keychain, Fortress Agent vault                        | git, AI, CI plaintext              |
| GitHub Actions `GITHUB_TOKEN`                                    | CI runtime only                                                  | persisted anywhere                 |

### 2.3 Trust zones

```text
   Zone 0 — Air-gapped vault         (paper + offline encrypted USB)
   Zone 1 — Fortress Agent runtime    (LXC 104 on N95, tmpfs decrypts)
   Zone 2 — Home LAN (Tailscale + Caddy + correct VLAN)
   Zone 3 — Dev laptop with anonymized clone
   Zone 4 — GitHub repo + Actions runners
   Zone 5 — Public internet
```

* Secrets flow Zone 0 → Zone 1 only, manually, via TOTP-gated unlock.
* Anonymized configs flow Zone 1 ↔ Zone 4 ↔ Zone 3.
* Apply flow is Zone 1 → Zone 2 (target HTTPS APIs).
* Zone 4 (GitHub) **never** holds a secret usable to log into anything.

```text
            │                                                          │
            │   OPNsense REST API   Proxmox API   Omada API            │
            │   (VLAN 40 ADMIN)     (VLAN 100)    (VLAN 10 MGMT)       │
            │                                                          │
            └──────────────────────────────────────────────────────────┘
```

**Network constraints applied to the agent:**

* Lives on VLAN 10 (`192.168.10.7/24`, MAC randomised at create).
* Holds **two** Tailscale identities, kept logically separate:
  * `tag:reader` — used for daily pulls, scoped by Tailscale ACL to OPNsense `:443`, Proxmox `:8006`, Omada `:8043` *read* endpoints only.
  * `tag:applier` — used **only** during a TOTP-approved apply window, scoped to the same hosts plus the *write* endpoints. Token leases are minted at unlock time and revoked when the window closes.
* Outbound internet allowed only to `api.github.com`, `*.snyk.io`, `*.tailscale.com`, `pkgs.tailscale.com`, `dl.cloudsmith.io`, plus the OS upgrade origins. Anything else is denied at the Proxmox cluster firewall *and* the LXC's own UFW.

### 4.1 Why this split

* **`docs/`** — every artefact is documented before it ships. ADRs capture *why*; runbooks capture *how*; research/architecture capture *what*.
* **`infra/`** — the *desired state*. Exports under `*/exports/` are the **anonymized** snapshots from `pull` operations. Deltas under `*/deltas/` are human-authored Jinja fragments composed onto the export to produce the next desired state. The agent re-hydrates and applies them.
* **`tools/`** — the machinery (anonymizer, agent, adapters, pre-commit). All Python because Python is already on every Proxmox / LXC and has good libs for OPNsense / Proxmox / Omada APIs.
* **`audit/`** — append-only artefacts from CI. Lets the user *see* whether a template ever passed, when it was last scanned, and what changed.
* **`.github/`** — CI is non-negotiable but minimal. No deploy from CI. CI never holds production credentials.

## 5. Anonymization pipeline

### 5.1 Goals (recap)

* **Deterministic.** The same secret in two files produces the same placeholder, so cross-file references survive (e.g. an alias name in `config.xml` matches the same alias name in a delta).
* **Reversible only by the agent.** Re-hydration requires `anonmap.age`, which lives only on the Fortress Agent (and the offline vault).
* **Schema-aware.** Anonymizing OPNsense `config.xml` knows about `<password>`, `<apikey>`, `<staticmap>/<mac>` etc., so it never accidentally mangles a tunable value that happens to look like a MAC.
* **Idempotent.** Running the anonymizer on an already-anonymized file is a no-op.

### 5.2 Components

* **`tools/anonymizer/rules.yaml`** — declarative rule set.
  * `regex_rules` — generic patterns (MAC, public IPv4, IPv6 GUA, age public key, JWT, SOPS-encrypted blob, Tailscale tailnet name, etc.).
  * `format_hooks` — format-specific Python callbacks for known schemas:
    * `opnsense_xml` — XPath-based scrubbing of `<password>`, `<apikey>`, `<apisecret>`, `<staticmap>/<mac>`, `<wireguard>/<key>`, `<wireguard>/<psk>`, `<wireguard>/<publickey>`, `<wan>/<mac>`, `<system>/<hostname>`, `<system>/<domain>`, `<openvpn>/*key*`, `<ipsec>/*psk*`, `<ssh>/<authorized_keys>`, `<captiveportal>/<users>/<user>/<password>`.
    * `proxmox_tfacfg` — masks TOTP secrets / WebAuthn descriptors in `/etc/pve/priv/tfa.cfg`.
    * `proxmox_clusterfw` — preserves rule semantics, scrubs comment-embedded notes that may leak hostnames.
    * `omada_db` — special handling of MongoDB exports: device MAC, AP serial, RADIUS PSK.
    * `caddy` — removes resolved upstream IPs from logs, preserves directives.
* **`tools/anonymizer/lexicon.example.yaml`** — committed example. Real lexicon is `lexicon.yaml` (gitignored).
* **`anonmap.age`** — the bidirectional placeholder ↔ secret map. Encrypted with the user's `age` recipient. Decrypted only on the Fortress Agent into a tmpfs (`/run/fortress/anonmap.json`, mode `0600`, owner `fortress`).
* **CLI:**
  * `tools/anonymizer/anonymize.py --in <file> --out <file> --format opnsense_xml` — produces an anonymized copy and *appends* to `anonmap.age`.
  * `tools/anonymizer/rehydrate.py --in <anonymized> --out <plaintext>` — agent-only.
  * `tools/anonymizer/anonymize.py --verify <path>` — invariant check, fails CI if the file contains anything that *looks* like a real secret.

### 5.3 Placeholder grammar

| Class                            | Placeholder format                                   | Examples                             |
| -------------------------------- | ---------------------------------------------------- | ------------------------------------ |
| MAC address                      | `MAC_<n>`                                            | `MAC_001`, `MAC_002`                 |
| Public IPv4                      | `PUBLIC_IPV4_<n>`                                    | `PUBLIC_IPV4_001`                    |
| Public IPv6                      | `PUBLIC_IPV6_<n>`                                    | `PUBLIC_IPV6_001`                    |
| Tailscale tailnet                | `TAILNET`                                            | (single value across the deployment) |
| Tailscale node FQDN              | `TS_NODE_<n>.TAILNET.ts.net`                         |                                      |
| Tailscale auth key               | `TS_AUTHKEY_<n>`                                     |                                      |
| OPNsense user (non-root)         | `OPNSENSE_USER_<n>`                                  |                                      |
| OPNsense password hash           | `OPNSENSE_PWD_HASH_<n>`                              |                                      |
| OPNsense API key                 | `OPNSENSE_API_KEY_<n>`                               |                                      |
| OPNsense API secret              | `OPNSENSE_API_SEC_<n>`                               |                                      |
| WireGuard private key            | `WG_PRIV_<n>`                                        |                                      |
| WireGuard public key             | `WG_PUB_<n>`                                         |                                      |
| WireGuard pre-shared key         | `WG_PSK_<n>`                                         |                                      |
| Proxmox API token                | `PVE_TOKEN_<n>`                                      |                                      |
| Proxmox TOTP secret              | `PVE_TOTP_<n>`                                       |                                      |
| Omada admin token                | `OMADA_TOKEN_<n>`                                    |                                      |
| Internal hostname                | `HOST_<n>` (or canonical role name when public-safe) |                                      |
| Internal device serial           | `SERIAL_<n>`                                         |                                      |
| TLS private key (PEM)            | `TLS_PRIV_<n>`                                       |                                      |

The map enforces **uniqueness per class**: `MAC_001` always means the same MAC. Adding a new MAC produces `MAC_<next>` and never reuses a number. The map is append-only; replacing a hardware MAC creates a new placeholder and marks the old one `retired: true`.

### 5.4 Pre-commit gate (Zone 3)

`tools/precommit/detect-plaintext-secrets.sh` runs on the dev laptop:

1. `gitleaks detect --no-banner --redact --staged`.
2. `python tools/precommit/verify-anonymization.py` — re-runs the anonymizer in `--verify` mode against every staged file.
3. Reject the commit on any hit.

The same pair runs in `.github/workflows/anonymization-gate.yml` so a missing pre-commit hook cannot bypass the gate.

### 5.5 What the dev laptop never has

* `anonmap.age` private key.
* `lexicon.yaml` (the populated lexicon).
* Production API tokens, even read-only.

The dev laptop holds only:

* The repository (anonymized).
* `lexicon.example.yaml` (placeholders only).
* A Snyk read-only token in the local sandbox.

## 6. Fortress Agent

### 6.1 Hosting

* **LXC 104** on `pve-n95` (so the agent is *not* co-located with the OPNsense VM on N305 — destroying N305 doesn't lose the agent).
* Image: `IMG-DEB12-HARDENED-v1` (no Docker — keep blast radius small).
* Resources: 1 vCPU, 1 GiB RAM, 4 GiB disk. Ephemeral state on tmpfs.
* Network: VLAN 10, IP `192.168.10.7/24`, GW `192.168.10.1`. UFW deny by default; allow only `api.github.com:443`, the OPNsense / Proxmox / Omada management IPs on their HTTPS ports, plus `*.snyk.io:443`.
* Tailscale node, tag `tag:fortress`. ACLs:
  * `tag:fortress` → `tag:server` on TCP 443, 8006, 8043, 8088, 8843, 9000.
  * `tag:fortress` → internet allow-list above.
  * No incoming connections accepted (default deny).

### 6.2 Vault & startup

```text
/run/fortress/                          (tmpfs, mounted at boot)
├── anonmap.json                        (rehydrated from anonmap.age at unlock)
├── api-tokens/
│   ├── opnsense-read.env               (read-only key+secret)
│   ├── opnsense-write.env              (only present during apply window)
│   ├── proxmox-n95-read.env
│   ├── proxmox-n95-write.env
│   ├── proxmox-n305-read.env
│   ├── proxmox-n305-write.env
│   ├── omada-read.env
│   └── omada-write.env
└── github-pat.env                      (fine-grained, this repo only)

/etc/fortress/
├── recipients.txt                      (age public keys allowed to unlock vault)
├── agent.toml                          (poll interval, target endpoints, label names)
└── anonmap.age                         (encrypted; loaded once at unlock)
```

* **Unlock** is manual: the user runs `fortress-unlock` on the agent (via Tailscale SSH from a hardened admin host on VLAN 40 ADMIN) with a YubiKey-backed `age` identity and a TOTP. Plaintext lives only in tmpfs and only for the unlock window (default 1 h, configurable per apply).
* **No password stored on the agent disk.** Reboot → agent boots into degraded mode (read-only pulls disabled, applies disabled) until the user unlocks it again.

### 6.3 Daemon behaviour

State machine:

```text
   idle ──poll(60s)──▶ check_open_prs ──┐
     ▲                                  │
     │                                  ▼
     │                         label "deploy:dry-run" present?
     │                                  │
     │                                  ├──yes──▶ rehydrate(branch) → validate → post audit comment
     │                                  └──no ──▶ continue
     │                                  ▼
     │                         label "deploy:apply" present?
     │                                  │
     │                                  ├──yes──▶ require_TOTP_via_webhook
     │                                  │           │
     │                                  │           ├── timeout/abort──▶ comment "apply rejected"
     │                                  │           └── confirmed ─────▶ apply → comment digest
     │                                  └──no ──▶ continue
     └──────────────────────────────────┘
```

* **Idempotent.** Each apply has an `apply_id` (UUIDv7). Re-applying the same `apply_id` is a no-op.
* **Atomic per target.** The OPNsense API supports config staging + commit; the agent stages, validates, then commits. Failure rolls back to the last known good `config.xml` from `infra/opnsense/exports/`.
* **Backups before every apply.** Agent calls each target's backup endpoint *first*, stores the backup under `/run/fortress/backups/<apply_id>/` and uploads an age-encrypted copy to the off-host backup share.

### 6.4 Audit trail

* Every operation appends a JSON line to `audit/agent-log/$(date +%Y-%m).jsonl` in the repo (committed by the agent on its own branch, then opened as a PR auto-merged into `audit-log`).
* Each line is signed with `minisign` using a key whose public part lives in `audit/keys/agent.pub`.
* On apply, the agent posts a PR comment containing:
  * Target endpoint (anonymized URL).
  * Diff summary (counts only — no secret leakage).
  * Snyk / Trivy / OPA gate hashes.
  * Apply duration, exit status, rollback availability.

## 7. Target adapters (no SSH)

### 7.1 OPNsense

* **API.** `https://opnsense.<TAILNET>.ts.net/api/` — key+secret via HTTP basic.
* **Privileges:**
  * `fortress-pull` user — group `Backup API` only. Can call `/api/core/backup/download/this`, `/api/core/firmware/info`, read-only inspectors. Cannot mutate.
  * `fortress-apply` user — group `Config Apply` only. Can call `/api/core/system/uploadconfig`, `/api/firewall/filter/apply`, `/api/wireguard/server/reconfigure`. Cannot read user passwords.
* **Pull flow:** download `config.xml` → anonymize → diff against `infra/opnsense/exports/config.anonymized.xml` → if changed, open a PR titled `chore(opnsense): drift from $TARGET_CONFIG_DIGEST`.
* **Apply flow:** start from current export → apply Jinja deltas → re-hydrate → upload via `uploadconfig` → call `apply` → fetch the new config to confirm digest match → re-anonymize → commit the new export back via PR.
* **Plugin:** [`os-api-backup`](https://docs.opnsense.org/manual/) for download; standard core APIs for upload/apply.

### 7.2 Proxmox VE (N95 + N305)

* **API.** `https://pve-{n95,n305}.<TAILNET>.ts.net:8006/api2/json/` — API token via header `Authorization: PVEAPIToken=…`.
* **Tokens:**
  * `fortress@pve!pull` — privileges `Sys.Audit`, `VM.Audit`, `Datastore.Audit`, `SDN.Audit`. Read-only.
  * `fortress@pve!apply` — privileges minimum needed per category (e.g. `Sys.Modify` for sysctl/firewall, `VM.Allocate` + `VM.Config.*` for LXC config, never `Sys.Console`).
* **What we manage via API:**
  * Cluster firewall (`/cluster/firewall/options`, `/cluster/firewall/rules`, `/cluster/firewall/aliases`, `/cluster/firewall/groups`, `/cluster/firewall/ipset`).
  * Storage definitions (`/storage`).
  * LXC create/update (`/nodes/{node}/lxc`).
  * LXC start/stop/restart (`/nodes/{node}/lxc/{vmid}/status/{start,stop,reboot}`).
  * LXC resize (`/nodes/{node}/lxc/{vmid}/resize`).
  * LXC config (`/nodes/{node}/lxc/{vmid}/config`).
  * 2FA configuration (`/access/users/{userid}/tfa` — read-only audit only).
* **What we cannot manage via API and how we work around it:**
  * `/etc/network/interfaces` — Proxmox exposes the SDN API and `/nodes/{node}/network` for bridge/VLAN edits. Use those, not file uploads.
  * `/etc/sysctl.d/*` and `/etc/fail2ban/*` — apply via the LXC-style "container running on the host" pattern: the agent ships a one-shot LXC-104 sibling task that runs `pve-tools-apply` via the Proxmox `pveexec` API on a dedicated *unprivileged* command queue. No long-running shell.
  * Where no API exists at all (rare), the agent uses the Proxmox **`pvesh` over the API** technique: `POST /nodes/{node}/exec` with a single, allow-listed, signed command bundle whose hash is reviewed in the PR. This is the *only* place the agent has remote command execution and it is gated by:
    * Allow-list of bundle hashes baked into `tools/fortress-agent/adapters/proxmox.py`.
    * TOTP confirmation per apply.
    * One-shot, no-tty, output captured into the audit trail.

### 7.3 Omada controller

* **API.** `https://omada.<TAILNET>.ts.net:8043/api/v6/...` — login via `/api/v2/login`, controllerId in path.
* **What we manage:**
  * Sites (`/sites`), Wired Networks (`/sites/{id}/setting/wired/networks`).
  * Wireless networks (`/sites/{id}/setting/wireless/wlans`).
  * Port profiles (`/sites/{id}/setting/wired/profiles`).
  * Switch port overrides (`/sites/{id}/eaps/{mac}/...` and `/sites/{id}/switches/{mac}/ports`).
  * Backups (`/controller/backup`).
* **Pull flow:** call `/controller/backup` → receive `.cfg` → anonymize → commit under `infra/omada/exports/`.
* **Apply flow:** PUT/POST the relevant endpoints; backup before, verify after.

### 7.4 Unifying interface

```python
# tools/fortress-agent/adapters/base.py
class TargetAdapter(Protocol):
    name: str
    def pull_config(self) -> bytes: ...
    def stage(self, payload: bytes, *, apply_id: str) -> StagingHandle: ...
    def apply(self, handle: StagingHandle) -> ApplyResult: ...
    def rollback(self, handle: StagingHandle) -> RollbackResult: ...
    def backup(self) -> Path: ...
```

Every adapter implements `pull_config`, `stage`, `apply`, `rollback`, `backup`. The agent's apply loop is target-agnostic.

## 8. Deploy flow (PR → apply)

### 8.1 Standard change

1. **Author** — dev opens a feature branch on the dev laptop (anonymized clone). Edits an export or a delta. Runs the local pre-commit hook, which calls the anonymizer in `--verify` mode.
2. **Push** — push to GitHub. CI runs:
   * `ci.yml` — yamllint, jsonlint, ansible-lint, terraform validate, `xmllint --noout`, OPA `secrets.rego`.
   * `snyk.yml` — `snyk code test`, `snyk iac test infra/**`, `snyk container test` for any LXC image touched.
   * `trivy.yml` — `trivy fs .` and `trivy config .`.
   * `anonymization-gate.yml` — `verify-anonymization.py` re-scan; commit must reproduce zero plaintext-secret findings.
3. **Open PR** — PR template (see §8.4) requires the dev to declare *which targets* will be touched and *what blast radius* is expected.
4. **Review** — the user reviews the diff, especially the "diff in cleartext after re-hydration" comment posted by the agent in `dry-run` mode.
5. **Dry-run** — user adds the `deploy:dry-run` label. Agent re-hydrates the branch, runs adapter-side `validate` (`opnsense-cli config-test`, `pveum role-spec dry-run`, etc.) and posts a structured comment.
6. **Apply** — user adds the `deploy:apply` label. Agent posts a TOTP challenge URL on its own webhook (`fortress.<TAILNET>.ts.net/totp/<apply_id>`). User opens it, submits TOTP. Agent applies, posts result, removes the label.
7. **Squash-merge** — once apply succeeds, the PR auto-receives an "applied" status check. User squash-merges.
8. **Drift check** — within 5 minutes the agent runs a fresh pull. If the cleartext result anonymizes to a digest equal to the new export, the change is recorded as `confirmed`. Otherwise the agent opens a `chore(drift)` PR.

### 8.2 Emergency (break-glass) change

* User pushes a branch with the `urgency:break-glass` label.
* Skips most CI checks **except** `anonymization-gate.yml` and OPA `secrets.rego`. These two are non-skippable.
* Still requires the TOTP confirmation. There is no path to apply without TOTP.

### 8.3 Rollback

* Each apply is paired with a backup taken under `apply_id`. Rollback = "apply the *previous* `config.xml`" with a fresh `apply_id`. The agent supports a one-click rollback PR comment: `/fortress rollback <apply_id>`.

### 8.4 PR template (`.github/pull_request_template.md`)

```text
## Summary
- What changes, in plain English.
- Which sections of the bible are affected.

## Targets touched
- [ ] OPNsense
- [ ] Proxmox N95
- [ ] Proxmox N305
- [ ] Omada controller
- [ ] LXC image / container
- [ ] Caddy / reverse proxy
- [ ] None (docs only)

## Blast radius
- [ ] Read-only (audit, docs)
- [ ] Cosmetic (no traffic effect)
- [ ] Network-affecting (firewall, DHCP, VLAN, DNS)
- [ ] Identity-affecting (users, 2FA, API tokens)
- [ ] Cryptographic (keys, certs, WireGuard)

## Anonymization
- [ ] No new placeholder classes introduced
- [ ] If new placeholders: rules.yaml updated AND lexicon.example.yaml updated

## Audit
- [ ] Snyk green
- [ ] Trivy green
- [ ] OPA green
- [ ] Anonymization gate green

## Rollback plan
- (How will we revert this if it goes wrong?)
```

## 9. Security audit & hardening pipeline

### 9.1 Snyk

* **Where keys live:** `.sandbox/state/snyk/.snyk-token` (gitignored) and a GitHub Actions secret `SNYK_TOKEN`.
* **What we run on every PR:**
  * `snyk auth $SNYK_TOKEN`.
  * `snyk code test` — SAST on Python under `tools/`.
  * `snyk iac test infra/` — Terraform / Ansible / Kubernetes manifests.
  * `snyk container test <image>` — for any built LXC image (we tag the LXC OCI export and feed it to Snyk).
* **Severity gate:** any **High** or **Critical** finding fails CI. **Medium** findings are tracked under `audit/snyk/findings.md` with an SLA.

### 9.2 Trivy

* `trivy fs .` — file-system scan for known-bad files / configs.
* `trivy config .` — IaC misconfig scan (Dockerfile, Terraform, Kubernetes).
* `trivy image <image>` — container image scan, complementing Snyk.

### 9.3 Lynis

* Run on each Proxmox host weekly via the agent (read-only audit, scheduled task in Proxmox cron). Output committed to `audit/lynis/`.
* Score below `80` opens a `chore(hardening)` PR pre-populated with the findings.

### 9.4 OPA / Conftest

* `infra/policies/secrets.rego` — forbid any string matching the placeholder patterns *unless* it is anonymized (this catches accidental cleartext that gitleaks missed).
* `infra/policies/opnsense.rego` — invariants:
  * No interface-level allow rule with `source = any` outside the temp-migration window (which must be tagged `temp-migration: true`).
  * `60_QUARANTINE` always has the forced-DNS rules.
  * `40_ADMIN` has no internet-egress rule.
  * No SSH service enabled with a wildcard listen interface.
  * Captive portal `EmergencyAccess` zone exists with `concurrentlogins: 1`.
* `infra/policies/proxmox.rego`:
  * Cluster firewall is `enabled: 1`.
  * Both `ssh.socket` and `ssh.service` are masked on all nodes.
  * 2FA is enrolled for the admin user.
  * No LXC has `unprivileged: 0` unless explicitly allow-listed in `proxmox-priv-allowlist.yaml`.
* `infra/policies/lxc.rego`:
  * Every LXC declares an audit-script path that exists.
  * `IMG-DEB12-HARDENED-v1` lineage check — every CT not on `DOCKER-v2` must inherit from HARDENED-v1.
  * `bridge-vlan-aware` must be `yes` on `vmbr0` whenever any CT references a VLAN tag > 1.

### 9.5 Pentest

* Quarterly: clone the LXC under test to a sandbox VM on N305 with no upstream connectivity, point an attacker LXC at it (`metasploitable`-style), and run an automated battery (`nuclei`, `nikto`, `gobuster`, `nmap` scripts).
* Findings checked into `audit/pentest/<date>/` and tagged into `audit/issues.md`.

### 9.6 Promotion gate

A LXC image becomes "golden" only when:

* Snyk green.
* Trivy green.
* Lynis ≥ 80.
* No open Critical from the latest pentest.
* Has an audit script (`audit_*.sh`) and a `verify_security.sh` ("Live Fire") script.
* OPA `lxc.rego` green.

## 10. Branch protection & approval gates

### 10.1 GitHub-side

* `master` is protected:
  * Required reviews: 1 (the owner — but PRs from the owner still need an explicit approval click on a *different* device).
  * Required status checks (all must be green): `ci`, `snyk`, `trivy`, `anonymization-gate`, `opa`.
  * Linear history.
  * Admins included.
* `feature/**` is the standard work branch. No protection.
* `audit-log` is protected with a single required check (`audit-log-signature-verify`) — only the agent should push there.

### 10.2 Repo-level

* `CODEOWNERS` requires the owner's review on every directory.
* Tag-based protection: only signed tags (`gpg`-signed by the owner) can be applied to `master`.

### 10.3 Agent-side

* No GitHub label by itself triggers an apply. The label arms the agent; TOTP is what ships the change.
* The TOTP secret is not the same as the user's Proxmox 2FA — separation of duties.
* Window: TOTP unlocks a 10-minute window. After the window, the apply must be re-armed (re-add label).
* All applies are logged with `apply_id` + signature; replay is impossible.

## 11. Worked example — adding AdGuard on N305

This is the canonical demo and the reason §1.1 says "DevOps-without-the-DevOps-team".

### 11.1 Goal

* New LXC `ct-105-adguard-dmz` on N305 (not N95) so its traffic to OPNsense stays *inside the CPU* (vmbr2, no physical NIC).
* Reachable through Caddy on LXC 100 N95 at `https://adguard.<TAILNET>.ts.net/`.
* DNS path: client → OPNsense Unbound → AdGuard (`vmbr2`) → Cloudflare DoT.
* Zero-Trust preserved: AdGuard cannot reach RFC1918 except OPNsense; cannot reach the internet except Cloudflare DoT and OS update origins.

### 11.2 Files the PR creates

```text
infra/lxc/ct-105-adguard-dmz/
├── README.md                               # what it does, why on N305
├── ct.yaml                                 # Proxmox API payload (declarative LXC spec)
├── compose.yaml                            # AdGuard Home docker-compose
├── adguard-config.yaml                     # AdGuard configuration template
└── audit_ct105.sh                          # audit script

infra/opnsense/deltas/
├── 2026-05-XX-vmbr2-dmz-interface.xml.j2   # adds vtnet2 + opt2-DMZ + rules
└── 2026-05-XX-vmbr2-dns-failover.xml.j2    # Unbound forward + serve-expired

infra/caddy/Caddyfile.j2                    # adds the adguard.<TAILNET>.ts.net vhost

infra/proxmox/n305/ansible/playbooks/
└── add-vmbr2.yaml                          # adds vmbr2 (no physical port) to /etc/network/interfaces via SDN API

audit/snyk/                                 # auto-generated reports for the PR
```

### 11.3 What runs in CI

* `snyk container test` against the AdGuard image (built into a tarball and scanned).
* `trivy config infra/lxc/ct-105-adguard-dmz/` — checks compose / yaml hardening.
* `opa eval` over `lxc.rego` — confirms HARDENED-v1 lineage, audit script presence, no privileged flag.
* `verify-anonymization.py` — confirms no leaked tailnet, MAC, IP.

### 11.4 What the agent does on `deploy:apply`

1. Re-hydrate `infra/opnsense/exports/config.anonymized.xml` + apply the two deltas → produce `config.xml`.
2. Backup current OPNsense config.
3. Upload + apply the new `config.xml` via `os-api-backup`.
4. Backup N305 LVM-thin pool snapshot of `local-lvm`.
5. Call Proxmox API `POST /nodes/pve-n305/lxc` with the rendered `ct.yaml` payload (includes `bridge: vmbr2`, `unprivileged: 1`, `features: nesting=1`, golden image base).
6. Call Proxmox API `POST /nodes/pve-n305/lxc/105/status/start`.
7. Run the LXC init script (via Proxmox `exec` API + signed bundle hash) which `cp` the AdGuard config into place and starts the compose stack.
8. Update Caddy on N95 LXC 100: PUT the rendered Caddyfile via the Caddy admin API (`localhost:2019/load`) — Caddy reload is hot, no downtime.
9. Verify: `curl --resolve adguard.<TAILNET>.ts.net:443:<LXC100_IP>` from the agent → expect `200 OK`.
10. Post the diff comment to the PR, remove the apply label, capture the audit log line.

### 11.5 What the user sees

* In the GitHub UI: the PR has a green "applied" check, a comment listing the targets touched, the backup IDs, and a one-click rollback hint.
* On the laptop: opening `https://adguard.<TAILNET>.ts.net/` over Tailscale just works.

### 11.6 What never happens

* No SSH session to N305.
* No plaintext secret leaves the agent's tmpfs.
* No rule with `source: any → destination: any` reaches the firewall.
* No image landed on a target that did not pass Snyk + Trivy.
